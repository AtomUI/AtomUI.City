# 文档先行治理规范

版本：v0.1
状态：强制执行
适用范围：AtomUI.City 所有架构设计、模块设计、代码实现、测试、构建、模板和文档改动

## 1. 核心原则

AtomUI.City 采用文档先行的研发规则。

强约束：

- 所有模块必须有完善的文档后，才能开始写代码。
- 任何代码改动、包边界改动、生命周期改动、公共 API 改动、依赖改动、构建规则改动，都必须先完成文档对齐。
- 文档对齐必须经过确认后，才能进入实现。
- 未完成文档对齐的任务，只能做分析、讨论、调研和文档设计，不能开始实现。

这条规则优先级高于开发效率。AtomUI.City 是框架项目，框架的公共范式、生命周期、扩展点和包边界必须先被清晰表达，再进入代码。

## 2. 适用范围

以下改动必须先更新或确认文档：

- 新增模块。
- 修改模块职责。
- 修改包依赖方向。
- 修改生命周期阶段。
- 修改公共 API。
- 修改编程范式。
- 修改路由、状态、事件、数据、安全、插件等核心机制。
- 新增或升级关键依赖。
- 修改 Build、Cli、Templates、Testing 行为。
- 修改仓库结构、代码风格、版本、打包、发布流程。
- 修复会改变框架行为的 bug。

只允许跳过文档更新的情况：

- 纯拼写修复。
- 不改变语义的格式化。
- 明确无行为变化的注释修正。
- 临时诊断命令执行。

即使跳过文档更新，也必须说明原因。

## 3. 模块开工条件

一个模块进入代码实现前，至少需要具备以下文档：

- `docs/modules/<module>/overview.md`
- 模块职责和非职责。
- 模块依赖和禁止依赖。
- 生命周期接入点。
- 核心概念和公共抽象。
- 扩展点。
- 错误处理策略。
- 测试策略。
- 与其他模块的集成关系。

复杂模块必须继续拆分细节文档。例如：

- Core：`lifecycle.md`、`hosting.md`、`modularity.md`、`configuration.md`、`dependency-injection.md`
- Mvvm：`activation.md`、`commands.md`、`interactions.md`、`validation.md`
- Routing：`route-graph.md`、`navigation.md`、`guards.md`、`resolvers.md`
- PluginSystem：`lifecycle.md`、`loading.md`、`unloading.md`、`contributions.md`

只有 `overview.md` 不足以支撑复杂模块实现。

## 4. 改动流程

任何实现任务必须按以下顺序推进：

```text
理解需求
-> 找到受影响文档
-> 更新或新增设计文档
-> 检查文档一致性
-> 等待确认
-> 编写实现计划
-> 开始代码实现
-> 更新测试
-> 回查文档是否仍然匹配实现
```

如果实现过程中发现原设计不成立，必须暂停实现，先更新文档并重新确认。

## 5. 文档对齐检查

开始代码前必须检查：

- 是否已有对应模块文档。
- 模块职责是否清晰。
- 模块边界是否清晰。
- 生命周期接入点是否清晰。
- 依赖方向是否符合 `architecture/package-boundaries.md`。
- 编程模式是否符合 `architecture/programming-model.md`。
- 依赖引入是否符合 `architecture/dependency-strategy.md`。
- PluginSystem 相关改动是否考虑运行时加载、停用、卸载和贡献撤销。
- 是否需要新增 ADR。

完成代码后必须检查：

- 实现是否偏离文档。
- 公共 API 是否仍符合文档。
- 测试是否覆盖文档承诺。
- 文档是否需要同步修订。

## 6. ADR 要求

以下情况必须新增或更新 ADR：

- 改变核心架构方向。
- 改变包边界。
- 引入新的强制编程范式。
- 引入会长期存在的关键依赖。
- 改变生命周期、插件、路由、状态、事件总线等基础机制。
- 明确放弃一种重要备选方案。

ADR 放在 `docs/decisions/`。

## 7. 禁止事项

以下行为禁止：

- 没有模块设计文档就直接写模块代码。
- 先写代码，再事后补一份追认文档。
- 公共 API 未经文档确认就落地。
- 生命周期、插件、路由、状态等核心机制绕过文档讨论直接实现。
- 文档和实现不一致时继续推进代码。
- 将外部框架参考材料混入 AtomUI.City 正式架构文档。

## 8. 责任边界

设计阶段的交付物是文档。

实现阶段的交付物是代码、测试和同步后的文档。

任何任务如果文档未确认，只能停留在设计阶段。

## 9. 验证要求

每次提交前至少验证：

- Markdown 链接无断链。
- 文档无明显占位内容。
- `git diff --check` 无格式问题。
- 如果包含代码改动，必须运行对应构建或测试命令。

文档先行不是可选流程，而是 AtomUI.City 的工程治理规则。
