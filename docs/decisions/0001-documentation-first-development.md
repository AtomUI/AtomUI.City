# 0001 文档先行研发规则

状态：Accepted
日期：2026-06-10

## 背景

AtomUI.City 是应用框架，不是普通业务应用。框架的生命周期、模块边界、公共 API、编程范式、插件机制、路由、状态、事件和构建规则都会长期影响使用者。

如果先写代码再补文档，容易出现以下问题：

- 公共 API 缺少设计约束。
- 模块边界随实现漂移。
- 生命周期和插件卸载等核心机制难以统一。
- 文档无法反映真实框架范式。
- 后续实现者无法判断改动是否符合整体架构。

因此，AtomUI.City 需要把文档先行作为强制研发规则。

## 决策

AtomUI.City 采用文档先行研发规则。

所有模块必须有完善的模块文档后，才能开始代码实现。

任何代码改动、包边界改动、生命周期改动、公共 API 改动、依赖改动、构建规则改动，都必须先完成文档对齐并经过确认，然后才能开始实现。

具体执行规则见：

- [文档先行治理规范](../engineering/documentation-governance.md)

## 影响

正向影响：

- 架构决策可追踪。
- 模块边界更稳定。
- 公共 API 更一致。
- 生命周期和插件等核心机制不会被局部实现牵着走。
- 新成员可以通过文档理解框架范式。
- 实现前能更早发现设计冲突。

成本：

- 每个实现任务前需要先维护文档。
- 小改动也需要判断是否影响文档。
- 设计确认会增加前置时间。

该成本是可接受的，因为 AtomUI.City 的主要风险不是写代码速度，而是框架范式失控。

## 后续约束

- 新模块没有完善文档不得开工。
- 复杂模块不能只保留 `overview.md`，必须拆分细节文档。
- 实现过程中发现设计不成立时，必须暂停实现并先更新文档。
- 提交前必须确认文档和实现一致。
- ADR 目录用于记录会形成长期约束的关键决策。
